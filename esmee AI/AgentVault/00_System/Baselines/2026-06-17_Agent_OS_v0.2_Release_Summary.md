## 5. v0.3 推荐方向

v0.3 推荐目标：

> 建立真实多部门调度协议最小版。

原因：

- v0.2 已证明真实部门会话可以收到任务、独立执行、写 result。
- 现在仍依赖总控手动记住 thread id、任务状态和 result 路径。
- 下一步应把“部门线程注册、任务派发、结果回收、汇总判定”固化成可审计模板。

v0.3 不建议立即做复杂自动化；先做最小可维护结构。

## 6. v0.3 第一小步建议

建议下一小步创建：

```text
AgentVault/00_System/Department_Thread_Registry.md
```

内容包括：

- 部门名称。
- Codex thread id。
- 当前状态：idle / active / blocked / archived。
- 负责范围。
- 最近一次任务 ID。
- 结果写回约定。
- 禁止事项。

同时创建：

```text
AgentVault/50_Memory/Department_Handoffs/real-v0-3-dispatch-protocol_*.md
```

用于让安全部、记忆部、工程部真实审查这个调度协议。

## 7. v0.3 成功标准

v0.3 第一阶段成功标准：

- 总控能从一个登记表找到所有部门会话。
- 每次派发有唯一 task id。
- 每个部门都有 result path。
- 总控能判断哪些部门完成、哪些等待、哪些阻断。
- 所有结果能回写到 Obsidian。
- Git 提交仍遵守 `Repository_Policy.md`。

## 8. 当前下一步

推荐下一步：

> 创建 `Department_Thread_Registry.md`，把当前真实部门线程登记为 v0.3 调度基础。

这一步只需要总控办公室执行，不需要立即让各部门再次并发。
