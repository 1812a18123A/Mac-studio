# Department Workflow v0.2

本文件把 v0.1 的行动记忆链路升级为可复用的多部门协作模板。它不引入自动化调度系统，只定义每次任务如何判断、记录、执行和交接。

## 1. 部门职责

总控办公室：

- 判断主责部门和协作部门。
- 控制任务范围，保持小步行动。
- 汇总状态、下一步和用户简报。

工程部：

- 处理代码、工具、测试、Git、仓库结构和本地脚本。
- 优先复用现有工具，避免从零实现核心功能。
- 提交前检查 diff、测试和版本控制边界。

记忆部：

- 创建、更新、审计行动记忆。
- 维护 `AgentVault/50_Memory/Action_Logs/` 和部门交接包。
- 确保行动结果、失败原因和下一步被记录。

安全部：

- 判断权限、隐私、外发、删除、GUI 自动化和账号操作风险。
- 对高风险动作转入确认状态。
- 维护安全边界文件和禁止事项。

技能优化部：

- 维护可复用 skill、模板、示例和使用说明。
- 当任务反复失败或流程重复时，提出 skill 更新建议。
- 不在未经确认时修改全局 skill。

## 2. 标准任务循环

每个任务按以下顺序执行：

1. 判断主责部门和协作部门。
2. 判断是否需要协作窗口或部门交接。
3. 创建或更新行动记忆。
4. 列出要改什么、不能做什么、需要哪些资料。
5. 检查是否需要用户确认。
6. 执行一个最小行动。
7. 把实际结果写回行动记忆。
8. 更新状态为 `completed`、`failed` 或 `blocked`。
9. 给用户简短中文汇报。

## 3. 确认规则

以下动作必须先停下确认：

- 删除、覆盖、批量移动或批量重命名。
- 修改 `.obsidian/` 内容、插件或全局配置。
- 外发、上传、提交表单、发送消息或 push 未检查内容。
- 操作密钥、token、账号或隐私资料。
- 启动 GUI 自动化或使用鼠标键盘控制。

可以直接执行的低风险动作：

- 读取当前项目文件。
- 写入 AgentVault 行动记忆。
- 创建用户已确认范围内的单个 Markdown 文档。
- 运行已有本地测试和只读检查。

## 4. Git 工作流

默认 Git 流程：

1. 先检查 `git status --short --branch`。
2. 修改前说明目标文件。
3. 修改后检查 `git diff --stat` 和关键 diff。
4. 只 stage 本次任务文件。
5. 提交前检查 `git diff --cached --stat`。
6. push 前确认远端、分支和外发范围。

禁止：

- 使用破坏性 git 命令回滚用户改动。
- 用 `git add .` 纳入不确定文件。
- 把 `.obsidian/`、顶层私人笔记或密钥文件混入提交。

## 5. 部门交接模板

需要交接时，在 `AgentVault/50_Memory/Department_Handoffs/` 创建短文档，包含：

- 发起部门和接收部门。
- 任务 ID 和优先级。
- 相关行动记忆。
- 当前状态和已完成工作。
- 请求下一部门执行的最小动作。
- 明确禁止事项。
- 成功标准。
- 结果写回位置。

## 6. 完成标准

一个任务完成时必须满足：

- 行动记忆状态已收口。
- 用户能看懂实际做了什么和下一步。
- 新增或修改文件路径明确。
- 风险和未完成事项已记录。
- 若涉及 Git，提交范围清楚且没有私有资料混入。

## 7. v0.2 使用方式

从 v0.2 起，每次任务默认先使用此模板做轻量判断。若任务很小，可以只在行动记忆中简短记录；若跨部门、涉及权限或要入库，则必须完整走本文件流程。
